- Do not create pseudo-species like "protein + metabolite" as entity records.
- reactants/products = species that are consumed or produced (stoichiometric participants).

- Create a protein_complex ONLY when the text explicitly says "complex", "heterodimer", "holoenzyme", or names a multi-subunit structure.
- Single-protein entries should NEVER be wrapped in a complex unless the text forces it.
- If exact subunit membership is unknown, still create the complex but mark confidence < 1.0 and provenance: "inferred" in the inference stage.
- class must always be filled; if uncertain, use the parent type name (e.g., "compound" for a compound).
  Valid class values — compound: compound | cofactor | ion | element_collection;
  protein: protein | transporter; protein_complex: protein_complex; nucleic_acid: nucleic_acid.
  Valid class values for processes — reactions: biochemical_reaction | transport_reaction | interaction | subpathway | activation | inhibition | binding;
  interactions: activation | inhibition | binding | interaction.
- confidence for extracted objects = 1.0; for inferred objects = LLM-judged value between 0.0 and 1.0.
- provenance values: "extracted" (verbatim from text), "enriched" (from API lookup), "inferred" (LLM-reasoned).
- source_refs is an array of strings — verbatim evidence quotes or external IDs when available.
- When the same compound appears in two compartments, create TWO compound_locations entries pointing to
  different biological_states — do NOT create two separate compound entities.


EDUCATED GUESS POLICY

3. Set provenance to "inferred"
4. Add a brief note in source_refs explaining the basis (e.g., "name pattern suggests enzyme", "UniProt fuzzy match")
This is better than omitting the entity entirely. The QA system will flag low-confidence items for review.

COMPARTMENT RULE (MANDATORY — always apply when subcellular locations are present)
